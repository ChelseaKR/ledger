# Internationalization & localization status: gettext (EN/ES)

ledger renders its end-user-facing text in **English and Spanish** through GNU
**gettext** message catalogs, per
[`STANDARDS/INTERNATIONALIZATION-STANDARD.md`](https://github.com/ChelseaKR/STANDARDS/blob/main/INTERNATIONALIZATION-STANDARD.md)
§3/§4. This replaces the previous bespoke `_CATALOG`/`_CW_GLOSSES` Python dicts in
`ledger.i18n` (which had no extraction tooling, no plural handling, and no
key-parity check).

Declared: 2026-06-30 · Reviewer: ledger maintainers

---

## What is (and is not) translated

**Translated — the accessible browse server + contribution form.** The rendered
HTML surface ([`ledger.render`](../src/ledger/render.py),
[`ledger.server`](../src/ledger/server.py),
[`ledger.contribute`](../src/ledger/contribute.py)) is where ledger emits
end-user-facing, natural-language text: navigation and search chrome, the
collection overview, record pages and content-warning glosses, the steward console
and audit log, the consent/takedown status lookup, and the whole contribution write
path (form, "what a stranger sees" preview, confirmation, edit, and withdrawal, plus
the contributor-facing validation messages). Every such fixed string resolves
through the gettext seam and is rendered in the negotiated language.

**Not translated — operator/developer text (English-only by design).** The standard
scopes i18n to end-user text, not operator logs, so these are deliberately left in
English and carry **no** gettext calls:

- the CLI status/notice output and `argparse` help ([`ledger.cli`](../src/ledger/cli.py));
- the public-submission **moderation** output;
- PREMIS/Dublin Core preservation-metadata field text and structured event records;
- any diagnostic/status string not shown to an archive visitor or contributor.

Language **autonyms** (`language_name`: `English`, `Español`) are intentionally
*not* gettext-translated — an autonym is invariant across the UI language, so a
language picker always reads naturally to a native speaker.

## The seam (how it works)

- [`src/ledger/i18n.py`](../src/ledger/i18n.py) is the single migration seam.
  `get_translation(lang)` loads `messages.mo` for the language with
  `fallback=True` (an unknown tag or a missing `.mo` returns the English source
  text, never an error). `negotiate(accept_language, available)` implements the §6
  `Accept-Language` fallback chain (`<requested> → <primary subtag> → site
  default (en)`), honoring `q` weights.
- **The public API is unchanged**, so no render/server/contribute call site moved:
  `t(lang, key, **kw)`, `gloss_cw(lang, tag)`, `language_name(code)`, and
  `negotiate(...)` keep their signatures. Internally they now resolve through
  gettext instead of a hand-maintained dict. `t` looks a stable UI key up in a
  `{key: _("English source")}` map built from the loaded catalog, so the
  **source string is the English text itself** — extraction needs no key map.
- Catalogs live **inside the package** at
  `src/ledger/locales/{messages.pot, <lang>/LC_MESSAGES/messages.{po,mo}}`, so a
  checkout or an installed wheel resolves them with no separate install step.
- [`babel.cfg`](../babel.cfg) tells `pybabel extract` to scan the package sources.

### Plurals

Messages with genuine singular/plural forms use `ngettext` (full CLDR plural
categories, `nplurals=2` for en/es): the "edited N times" steward badge
(`badge_edited`) and the "N details are restricted" outsider notice
(`rec_withheld_outsider`). The two record-count display strings (`overview_total`,
`results_showing`) keep their established `record(s)` single-message form rather
than being split into `ngettext`: they are asserted verbatim across the
pagination/overview/server/browse tests, and converting them buys no
plural-correctness the `(s)` form does not already convey, while widening the blast
radius into the disclosure/server suites. This is a deliberate, recorded scope
decision, not an oversight.

## Spanish translation status — complete (no MT, no backlog)

Spanish is **fully translated**. ledger already shipped a **real EN/ES bilingual
surface**: `_CATALOG["es"]` and `_CW_GLOSSES["es"]` were complete, human-authored
Spanish, not an English-only-with-ES-owed stub. This migration is therefore a
**format migration of existing translations**, carried over **verbatim** into
`es/LC_MESSAGES/messages.po`. **No machine translation was introduced.**

Because real Spanish exists and is complete, the **G5 completeness gate is enforced
as a hard merge-blocker** (every msgid has a non-empty `msgstr` in every locale,
including every plural form) — *not* downgraded to a structural-only check with an
untranslated-ES follow-on. There is no
[REVIEW-GATE R3](https://github.com/ChelseaKR/STANDARDS/blob/main/INTERNATIONALIZATION-STANDARD.md#5-review-gates-human-judgment--committed-artifact)
translation backlog for the current string set. Any *new* user-facing string is
added in English first and must have its Spanish `msgstr` filled before merge (the
gates below block otherwise).

## Live merge-blocking gates (`make i18n`, wired into `make verify` and CI)

Local `make i18n` and the CI `i18n` job run the identical target (no drift):

| Gate | What it checks | Mechanism |
|---|---|---|
| **G2-lite** | The extraction template is committed & current | `pybabel extract` → normalize → `git diff --exit-code` on `messages.pot`. A new/changed user-facing string without a re-extract fails. |
| **G7** | Every PO compiles cleanly | `msgfmt --check --check-format --check-domain` on each `messages.po` |
| **G6** | EN/ES key-parity | [`tools/check_catalog_parity.py`](../tools/check_catalog_parity.py): msgid sets of `en` and `es` are identical (empty symmetric difference) |
| **G5** | Completeness + placeholder parity | same script: every `msgstr` (each plural form) non-empty; `{...}` field set identical source↔target |
| **G3** | BCP 47 / RFC 5646 tag validity | [`tools/check_bcp47.py`](../tools/check_bcp47.py): `babel.Locale.parse` on every authored locale tag |

`msgfmt` is a system **gettext** binary (like `gitleaks`, not a pip dependency); CI
installs it with `apt-get install -y gettext`. `pybabel`/`babel.Locale` come from
the `babel>=2.16` dev dependency (pure Python — no PyICU/native ICU build, so ledger
keeps its "run on one inexpensive box, no heavy dependency tree" property).

### The POT normalizer

`pybabel extract` emits volatile tokens (creation timestamp, Babel version, the
current year in the boilerplate comment) and would tag any literal-`%` string as a
false `python-format` (printf) message.
[`tools/i18n_normalize_pot.py`](../tools/i18n_normalize_pot.py) freezes those tokens
and drops the spurious flag so the G2-lite diff is meaningful, not flaky, and so
`msgfmt --check-format` never wrongly rejects a literal percent sign. The gate runs
the normalizer on both authoring and CI, keeping the committed POT byte-stable.

## Compiled `.mo` files are committed

The compiled `messages.mo` catalogs **are committed** (not gitignored). Rationale:
ledger's CI runs single `make`/`pytest` steps per job with no packaging/build step,
and the test suite renders the browse pages and asserts Spanish output, so the
catalogs must be present at runtime. Committing `.mo` keeps every job green with
local == CI and no extra per-job compile, and ships the translations in the wheel.
`.po` ↔ `.mo` drift is guarded end-to-end by the render/server tests (they render and
assert real Spanish); regenerate `.mo` after editing a `.po` with
**`make i18n-compile`**.

## Workflow: adding or changing a user-facing string

1. Add the string to the `_messages` (or `_cw_glosses`) map in
   [`src/ledger/i18n.py`](../src/ledger/i18n.py) as `"key": _("English source")`
   (or, for a count, add an `ngettext(...)` case to `_plural_message` and register
   the key in `_PLURAL_KEYS`). Use `.format()` placeholders (`_("… {n} …")`).
2. `make i18n` — re-extracts `messages.pot`; the G2-lite diff shows the new msgid.
3. Merge the new msgid into the catalogs (`pybabel update -i
   src/ledger/locales/messages.pot -d src/ledger/locales --no-fuzzy-matching`),
   then fill the English `msgstr` (= msgid) and the Spanish `msgstr`.
4. `make i18n-compile` to rebuild the `.mo`, then `make i18n` (and `make verify`).
5. Commit the changed `.py`, `messages.pot`, both `messages.po`, and both `messages.mo`.

## Scope not yet covered (honest boundaries)

This migration delivers the gettext catalog + the PO-specific gates
(G2-lite/G3/G5/G6/G7). Other standard gates are **not** implemented here and remain
future work: G1 (UTF-8 byte gate), G4 (`html-has-lang` — the browse pages already
set `lang`, but graduating the axe rule to a blocking i18n gate is separate), G9/G10
(pseudolocale/RTL — ledger ships no RTL locale), G11 (`Vary: Accept-Language` — the
browse server negotiates the language but does not yet emit the `Vary`/
`Content-Language` headers on every response), and G12 (CLDR/tzdata freshness pin).
CLDR data is whatever the pinned `babel` bundles; revisit on the next Babel/CLDR
major per the standard's cadence.
