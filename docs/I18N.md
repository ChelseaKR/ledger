# Internationalization & localization status: gettext (EN/ES/FR/AR, incl. RTL)

ledger renders its end-user-facing text in **English, Spanish, French, and Arabic**
through GNU **gettext** message catalogs, per
[`STANDARDS/INTERNATIONALIZATION-STANDARD.md`](https://github.com/ChelseaKR/STANDARDS/blob/main/INTERNATIONALIZATION-STANDARD.md)
§3/§4. Arabic is right-to-left, so the browse pages carry a base text direction
(`<html dir="rtl">`, see [RTL & text direction](#rtl--text-direction-g10)). This
replaces the previous bespoke `_CATALOG`/`_CW_GLOSSES` Python dicts in `ledger.i18n`
(which had no extraction tooling, no plural handling, and no key-parity check).

Declared: 2026-06-30 · Updated: 2026-07-02 (RM7: +fr/+ar, RTL, remaining gates)
· Reviewer: ledger maintainers

---

## What is (and is not) translated

**Translated — the accessible browse server + contribution form.** The rendered
HTML surface ([`ledger.render`](../src/ledger/render.py),
[`ledger.server`](../src/ledger/server.py),
[`ledger.contribute`](../src/ledger/contribute.py)) is where ledger emits
end-user-facing, natural-language text: navigation and search chrome, the
collection overview, record pages and content-warning glosses, the steward console
and audit log, the consent/takedown status lookup, and the whole contribution write
path (form, "what a stranger sees" preview, confirmation, edit, and withdrawal, plus
the contributor-facing validation messages). Every such fixed string resolves
through the gettext seam and is rendered in the negotiated language.

**Not translated — operator/developer text (English-only by design).** The standard
scopes i18n to end-user text, not operator logs, so these are deliberately left in
English and carry **no** gettext calls:

- the CLI status/notice output and `argparse` help ([`ledger.cli`](../src/ledger/cli.py));
- the public-submission **moderation** output;
- PREMIS/Dublin Core preservation-metadata field text and structured event records;
- any diagnostic/status string not shown to an archive visitor or contributor.

Language **autonyms** (`language_name`: `English`, `Español`, `Français`,
`العربية`) are intentionally *not* gettext-translated — an autonym is invariant
across the UI language, so a language picker always reads naturally to a native
speaker.

## RTL & text direction (G10)

`ledger.i18n.text_direction(lang)` returns `"rtl"` for a right-to-left script
(primary subtag in `{ar, he, fa, ur}`, matched case-insensitively so `ar-EG`
resolves) and `"ltr"` for everything else, including an unknown or malformed tag
(the safe default — a page never renders with an empty or wrong direction). The
shared page shell (`ledger.render._page`) threads this into the document root, so
every rendered page declares both the language and its direction:

```html
<html lang="ar" dir="rtl">   <!-- Arabic: right-to-left -->
<html lang="fr" dir="ltr">   <!-- French/English/Spanish: left-to-right -->
```

Because `dir` is set on `<html>`, the browser mirrors the whole layout for an RTL
reader with no per-element CSS. Adding another RTL language is just adding its
catalog and (if its primary subtag is new) one entry to `_RTL_LANGS`.

## The seam (how it works)

- [`src/ledger/i18n.py`](../src/ledger/i18n.py) is the single migration seam.
  `get_translation(lang)` loads `messages.mo` for the language with
  `fallback=True` (an unknown tag or a missing `.mo` returns the English source
  text, never an error). `negotiate(accept_language, available)` implements the §6
  `Accept-Language` fallback chain (`<requested> → <primary subtag> → site
  default (en)`), honoring `q` weights.
- **The public API is unchanged**, so no render/server/contribute call site moved:
  `t(lang, key, **kw)`, `gloss_cw(lang, tag)`, `language_name(code)`, and
  `negotiate(...)` keep their signatures. Internally they now resolve through
  gettext instead of a hand-maintained dict. `t` looks a stable UI key up in a
  `{key: _("English source")}` map built from the loaded catalog, so the
  **source string is the English text itself** — extraction needs no key map.
- Catalogs live **inside the package** at
  `src/ledger/locales/{messages.pot, <lang>/LC_MESSAGES/messages.{po,mo}}`, so a
  checkout or an installed wheel resolves them with no separate install step.
- [`babel.cfg`](../babel.cfg) tells `pybabel extract` to scan the package sources.

### Plurals

Messages with genuine singular/plural forms use `ngettext` (full CLDR plural
categories: `nplurals=2` for en/es/fr, `nplurals=6` for Arabic — the `pybabel init`
header carries the right rule and slot count per locale): the "edited N times"
steward badge
(`badge_edited`) and the "N details are restricted" outsider notice
(`rec_withheld_outsider`). The two record-count display strings (`overview_total`,
`results_showing`) keep their established `record(s)` single-message form rather
than being split into `ngettext`: they are asserted verbatim across the
pagination/overview/server/browse tests, and converting them buys no
plural-correctness the `(s)` form does not already convey, while widening the blast
radius into the disclosure/server suites. This is a deliberate, recorded scope
decision, not an oversight.

## Translation status — es/fr/ar all complete (no backlog)

**Spanish** is a **format migration of an existing translation**: `_CATALOG["es"]`
and `_CW_GLOSSES["es"]` were already complete, human-authored Spanish, carried over
**verbatim** into `es/LC_MESSAGES/messages.po`.

**French** (`fr/LC_MESSAGES/messages.po`) and **Arabic**
(`ar/LC_MESSAGES/messages.po`) were authored for RM7: every msgid (including both
plural cases, and Arabic's six CLDR plural forms) and every content-warning gloss is
translated, with `{…}` placeholders preserved so the placeholder-parity gate holds.

Because every shipped locale is complete, the **G5 completeness gate is enforced as
a hard merge-blocker** — every msgid has a non-empty `msgstr` in **every** locale
(en/es/fr/ar), including every plural form — *not* downgraded to a structural-only
check. There is no
[REVIEW-GATE R3](https://github.com/ChelseaKR/STANDARDS/blob/main/INTERNATIONALIZATION-STANDARD.md#5-review-gates-human-judgment--committed-artifact)
translation backlog for the current string set. Any *new* user-facing string is
added in English first and must have its `msgstr` filled in **all four** locales
before merge (the gates below block otherwise). Community translators are the
intended path for further review/refinement of the fr/ar strings.

## Live merge-blocking gates (`make i18n`, wired into `make verify` and CI)

Local `make i18n` and the CI `i18n` job run the identical target (no drift):

| Gate | What it checks | Mechanism |
|---|---|---|
| **G1** | UTF-8 end to end | [`tools/check_i18n_utf8.py`](../tools/check_i18n_utf8.py): every `.po`/`.pot` decodes as strict UTF-8 and declares `charset=utf-8`; a page rendered in each locale round-trips through UTF-8 and declares `<meta charset="utf-8">`. |
| **G2-lite** | The extraction template is committed & current | `pybabel extract` → normalize → `git diff --exit-code` on `messages.pot`. A new/changed user-facing string without a re-extract fails. |
| **G3** | BCP 47 / RFC 5646 tag validity | [`tools/check_bcp47.py`](../tools/check_bcp47.py): `babel.Locale.parse` on every authored locale tag |
| **G5** | Completeness + placeholder parity | [`tools/check_catalog_parity.py`](../tools/check_catalog_parity.py): every `msgstr` (each plural form, incl. Arabic's six) non-empty; `{...}` field set identical source↔target |
| **G6** | Key-parity across every locale | same script: msgid sets of `en`/`es`/`fr`/`ar` are each identical to the template (hence to one another) |
| **G7** | Every PO compiles cleanly | `msgfmt --check --check-format --check-domain` on each locale's `messages.po` |
| **G9** | Pseudolocale (test-only) | `i18n.pseudolocalize` accent-wraps a string keeping `{…}` intact; `tests/test_i18n_rtl.py` renders chrome through it and asserts no un-wrapped (hardcoded) English leaks. Never a shipped locale. |
| **G10** | RTL direction | `i18n.text_direction` → `<html lang dir>`; asserted for `ar` (rtl) vs en/es/fr (ltr). |
| **G11** | `Content-Language` + `Vary` | `ledger.server._send` emits `Content-Language: <negotiated>` and `Vary: Accept-Language` on **every** response; asserted across HTML/text/JSON routes. |
| **G12** | CLDR / locale-data freshness pin | [`tools/check_i18n_deps.py`](../tools/check_i18n_deps.py): Babel within the reviewed `>=2.16,<3` range and its bundled CLDR data loads (plural categories + display names resolve). |

`msgfmt` is a system **gettext** binary (like `gitleaks`, not a pip dependency); CI
installs it with `apt-get install -y gettext`. `pybabel`/`babel.Locale` come from
the `babel>=2.16,<3` dev dependency (pure Python — no PyICU/native ICU build, so
ledger keeps its "run on one inexpensive box, no heavy dependency tree" property).
The upper bound is the **G12** CLDR ceiling: a new Babel major bundles a new CLDR
release and must be adopted deliberately, not picked up silently.

### The POT normalizer

`pybabel extract` emits volatile tokens (creation timestamp, Babel version, the
current year in the boilerplate comment) and would tag any literal-`%` string as a
false `python-format` (printf) message.
[`tools/i18n_normalize_pot.py`](../tools/i18n_normalize_pot.py) freezes those tokens
and drops the spurious flag so the G2-lite diff is meaningful, not flaky, and so
`msgfmt --check-format` never wrongly rejects a literal percent sign. The gate runs
the normalizer on both authoring and CI, keeping the committed POT byte-stable.

## Compiled `.mo` files are committed

The compiled `messages.mo` catalogs **are committed** (not gitignored). Rationale:
ledger's CI runs single `make`/`pytest` steps per job with no packaging/build step,
and the test suite renders the browse pages and asserts translated output, so the
catalogs must be present at runtime. Committing `.mo` keeps every job green with
local == CI and no extra per-job compile, and ships the translations in the wheel.
`.po` ↔ `.mo` drift is guarded end-to-end by the render/server tests (they render and
assert real Spanish/French/Arabic); regenerate `.mo` for **all four** locales after
editing a `.po` with **`make i18n-compile`**.

## Workflow: adding or changing a user-facing string

1. Add the string to the `_messages` (or `_cw_glosses`) map in
   [`src/ledger/i18n.py`](../src/ledger/i18n.py) as `"key": _("English source")`
   (or, for a count, add an `ngettext(...)` case to `_plural_message` and register
   the key in `_PLURAL_KEYS`). Use `.format()` placeholders (`_("… {n} …")`).
2. `make i18n` — re-extracts `messages.pot`; the G2-lite diff shows the new msgid.
3. Merge the new msgid into every catalog (`pybabel update -i
   src/ledger/locales/messages.pot -d src/ledger/locales --no-fuzzy-matching`),
   then fill the `msgstr` in **all four** locales — English (= msgid), Spanish,
   French, and Arabic (all six plural forms for a count in `ar`). The G5 gate blocks
   an empty `msgstr` in any locale.
4. `make i18n-compile` to rebuild the `.mo`, then `make i18n` (and `make verify`).
5. Commit the changed `.py`, `messages.pot`, all four `messages.po`, and all four
   `messages.mo`.

### Workflow: adding a new UI language

1. `pybabel init -i src/ledger/locales/messages.pot -d src/ledger/locales -l <tag>`
   (this writes the correct `Plural-Forms` and slot count for the locale), then
   `pybabel update -i src/ledger/locales/messages.pot -d src/ledger/locales
   --no-fuzzy-matching` to keep it in step with the template.
2. Translate every `msgstr` (all plural forms), preserving `{…}` placeholders.
3. Add the tag to `SUPPORTED` and an autonym to `_LANGUAGE_NAMES` in `ledger.i18n`.
   If it is a **right-to-left** language whose primary subtag is not already in
   `_RTL_LANGS`, add it there so `text_direction` returns `"rtl"` and the page shell
   emits `<html dir="rtl">`.
4. `make i18n-compile`, then `make i18n && make verify`. The parity/completeness,
   UTF-8, BCP-47, and CLDR gates all pick the new locale up automatically (they
   iterate `SUPPORTED`).

## Scope now covered (RM7)

All i18n gates in the standard are now live in `make i18n`/`make verify`:
G1 (UTF-8), G2-lite (POT current), G3 (BCP-47), G5 (completeness/placeholder),
G6 (key-parity across en/es/fr/ar), G7 (PO compiles), G9 (test-only pseudolocale),
G10 (RTL `dir`), G11 (`Content-Language` + `Vary` on every response), and G12
(CLDR/Babel freshness pin). See the gates table above for each mechanism.

**G4** (`html-has-lang`) is covered by the accessibility gate rather than the i18n
gate: the browse pages set `lang` **and** `dir` on `<html>`, and
`make accessibility` (axe-style checks over the rendered surface) enforces it. CLDR
data remains whatever the pinned `babel` bundles; the G12 upper bound (`babel<3`)
makes bumping it a deliberate, reviewed step per the standard's cadence.
