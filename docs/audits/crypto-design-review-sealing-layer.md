# Crypto design review — the at-rest sealing layer (FIX-11)

## Status

**Proposed — internal design doc, external cryptographer review not yet
commissioned.** This document is the artifact meant to be *handed to* an external
reviewer, not a record that a review already happened. Per the FIX-11 remediation
item's own instruction, none of the changes proposed below may ship on
self-review; they are held open until a named, external cryptography reviewer
signs off (or amends them) in writing, and that sign-off is appended to this file
under [§6](#6-review-record).

- **Scope.** Only the *at-rest sealing* constructions in
  `src/ledger/identity.py`: the Fernet key(s) used to encrypt the identity vault
  and SEALED-tier content, the `enc:` string-prefix envelope, and the scrypt
  passphrase-derivation parameters.
- **Out of scope.** RM1's threshold/multi-party key custody (a distribution
  question, not a construction question) and RM9's general security audit (this
  repo's whole attack surface, not one primitive). This review is deliberately
  narrower and is meant to run *before* those, so it does not get bypassed by a
  broader audit that treats these choices as already settled.
- **Why now.** RM2 plans to broaden at-rest encryption beyond the identity vault
  and SEALED-tier content that use it today. Every additional caller that adopts
  today's `IdentityVault.encrypt_text`/`encrypt_bytes` shape widens the blast
  radius of any flaw here and raises the cost of changing the envelope later
  (more data would need re-encryption under a new format). This document exists
  to get the key hierarchy and envelope right *before* that fan-out, per
  `docs/ideation/02-large-scale-fixes.md`'s FIX-11 entry.

## 1. What exists today (the thing being reviewed)

`IdentityVault` (`src/ledger/identity.py`) holds a single Fernet key, supplied at
construction (`__init__`, generated by `generate_key()` or derived from a
passphrase by `derive_key()`), and uses that *one* key for three purposes:

1. **Identity ciphertext.** `add`/`resolve`/`rekey` encrypt/decrypt the
   `ContributorIdentity` records that back every `identity_ref` (`_encrypt`,
   `_decrypt`).
2. **Sealed field ciphertext.** `encrypt_text`/`decrypt_text`
   (`src/ledger/identity.py:293-311` in this revision) encrypt an absolute-SEALED
   field's value, called from `ingest.py:402`. The result is stored as the string
   `"enc:" + base64(fernet_token)`.
3. **Sealed payload ciphertext.** `encrypt_bytes`/`decrypt_bytes`
   (`src/ledger/identity.py:313-329`) encrypt an absolute-SEALED payload file's
   bytes wholesale, called from `ingest.py:330`. No prefix is used here because
   the caller never stores this as an in-band string — it goes straight to a
   content-addressed file.

Key derivation for a passphrase-unlocked vault: `derive_key` runs scrypt
(`N=2**15, r=8, p=1`, 32-byte output, RFC 7914 parameters) over a caller-supplied
salt and passphrase, then base64-encodes the result into a Fernet key.

## 2. What the ideation write-up flags, restated precisely

Two concrete concerns, both real and both fixable without a rethink of the
vault's overall architecture (identity separated from record, grant-gated
resolve — ADR 0003 — is not in question here):

### 2.1 One key protects two different asset classes

The vault key that decrypts *who a contributor is* is the same key that
decrypts *sealed content they contributed*. These have different compulsion and
threat profiles:

- An identity is compact, high-value, and — per `docs/THREAT-MODEL.md` §4.1/4.2
  — the single worst thing a compelled or seized key can expose (it outs a real
  person).
- Sealed content is often larger (oral-history transcripts, media files under
  FIX-03's chunked-sealing plan) and its exposure, while serious, is a different
  kind of harm and may need a different retention/rotation cadence than identity
  data (e.g. a steward may want to rotate content keys on a compliance cadence
  independent of any identity-vault event, or vice versa).
- `IdentityVault.rekey()` rotates identity ciphertext and sealed-content
  ciphertext *together*, because they share a key. There is no way today to
  rotate one without touching the other, and no way to reason about "how much
  sealed content is protected by a key that also protects identities" as a
  distinct exposure question.

### 2.2 The `enc:` prefix is an in-band, ambiguous type marker

`encrypt_text` stores ciphertext as a plain string beginning `"enc:"`. Nothing
stops a legitimate, unsealed field value from *also* beginning with those four
characters — the vault has no reserved-character enforcement on plaintext field
input. A future code path that treats any `"enc:"`-prefixed string as ciphertext
(a generic redaction sweep, a migration script, a new field type that also uses
this vault) would misparse that legitimate value, either failing to decrypt it
(denial of service on that field) or, worse, attempting to feed non-ciphertext
bytes through Fernet's authenticated decrypt and drawing the wrong conclusion
from the failure. The marker also carries no version: if the envelope ever
changes shape, `"enc:"` cannot express which shape a given token is.

## 3. Proposed key hierarchy (for the reviewer to bless or amend)

Replace "one Fernet key, three uses" with **one root key, purpose-bound
subkeys, derived, never stored separately**:

```
root vault key (unchanged inputs: generate_key() / derive_key())
  │
  ├── HKDF-SHA256(info=b"ledger:identity:v1")   -> identity key    (asset class: identity)
  ├── HKDF-SHA256(info=b"ledger:content-field:v1") -> content-field key (asset class: sealed field values)
  └── HKDF-SHA256(info=b"ledger:content-payload:v1") -> content-payload key (asset class: sealed payload bytes)
```

- The **root key is what a deployer generates, stores, and backs up** — the
  operational story in `docs/THREAT-MODEL.md` §4.1 (env var, keystore, hardware
  token) does not change, and there is still exactly one secret a steward must
  protect at rest.
- **Subkeys are derived on open, held only in memory, never persisted.** This
  keeps `IdentityVault.__init__`'s contract ("the key is held only to
  encrypt/decrypt and is never serialized or logged") intact for three keys
  instead of one.
- Each `info` label is a fixed, versioned domain-separation string, so a subkey
  for one purpose can never be confused with or reused as a subkey for another
  purpose even though they derive from the same root — this is the standard
  HKDF domain-separation argument, not a novel construction, which is
  deliberate: **the question for the reviewer is whether this specific
  application of it is sound**, not whether to invent something new.
- **Rotation becomes independent** *of the root* only if the root itself
  changes — rotating a single subkey in isolation (content-field key only,
  say) is *not* possible under pure HKDF-from-one-root without also changing
  what the root produces for the other two purposes. This is called out
  explicitly in [§5](#5-open-questions-for-the-reviewer) as the main design
  fork the reviewer should decide: HKDF subkeys (simpler, but rotation is still
  all-or-nothing at the root) versus genuinely independent stored subkeys, one
  per asset class (true independent rotation, at the cost of three secrets to
  protect instead of one).

## 4. Proposed envelope (for the reviewer to bless or amend)

Replace the bare `"enc:"` string prefix with an explicit, versioned,
structurally-distinct envelope carried *alongside* the ciphertext rather than
smuggled inside the same string slot as plaintext would occupy:

- **Version byte, not a string prefix.** A one-byte envelope version
  (`0x01` for this proposal) placed *before* base64-encoding, so the on-disk
  form is `base64(version_byte || key_class_byte || fernet_token_bytes)`. A
  legitimate plaintext field value can still coincidentally decode as valid
  base64, but it cannot coincidentally decode to a byte string whose first two
  bytes are a currently-defined `(version, key_class)` pair *and* whose
  remainder authenticates as a valid Fernet token under any of the three
  subkeys — Fernet's own HMAC makes a false-positive parse cryptographically
  implausible, which a bare string prefix does not give you.
- **`key_class` byte** records which of the three subkeys (§3) produced the
  token (`0x01` identity, `0x02` content-field, `0x03` content-payload), so a
  reader does not need to guess which subkey to attempt decryption with, and so
  the envelope self-documents which asset class a given ciphertext belongs to
  for audit purposes.
- **Config schema migration.** `Config.schema_version` (`src/ledger/config.py`)
  gains a bump when this lands, so an archive created under the old
  single-key/string-prefix scheme is detected on load and can be offered an
  explicit, logged re-encryption migration rather than silently misread — the
  same "upgrade or refuse a too-new file" contract `config.py`'s own docstring
  already commits to for its own schema.
- **Backward-read, forward-write.** During the migration window, `decrypt_text`
  should still accept the legacy `"enc:"` prefix on read (so an un-migrated
  archive keeps functioning) while `encrypt_text` always writes the new
  envelope — this is standard "read both, write one" migration practice and is
  called out so the reviewer can confirm it does not create a downgrade
  vulnerability (e.g. an attacker forcing a legacy-format write is not possible
  since only `encrypt_text` produces new ciphertext, and it never emits the old
  form).

## 5. Open questions for the reviewer

This is the actual ask — the numbered list a commissioned cryptographer should
answer, amend, or reject:

1. **Subkey derivation vs. independent stored keys** (§3). Is HKDF-derived,
   in-memory-only subkeying from one root an acceptable trade for this threat
   model, or does the "different compulsion profiles" concern in §2.1 require
   genuinely independent, independently-stored content key(s) — accepting the
   operational cost of a steward protecting more than one secret?
2. **HKDF construction details.** `HKDF-SHA256` with what salt (none, a
   per-vault random salt stored alongside the vault, or a fixed constant)? What
   `info` string format is actually domain-separating enough, and does content
   need one subkey or one *per asset class per rotation epoch*?
3. **Envelope soundness** (§4). Does the version-byte-plus-key-class-byte
   envelope actually eliminate the misparse risk described in §2.2, or does it
   merely shrink the collision space? Is there a cleaner approach the reviewer
   would prefer (e.g. never storing sealed ciphertext as an in-band string at
   all — always wrapping it in a schema-typed field so no string-sniffing is
   ever needed on read)?
4. **scrypt parameters.** Are `N=2**15, r=8, p=1` (the current
   `derive_key` parameters, `_SCRYPT_N`/`_SCRYPT_R`/`_SCRYPT_P` in
   `src/ledger/identity.py`) adequate for a key that, once derived, may protect
   years of accreted sealed content and identities — i.e. should this use
   higher, "sensitive data" parameters (OWASP's scrypt guidance suggests
   `N=2**17` or higher for long-lived, high-value secrets) rather than
   interactive-login-grade cost, given ledger's single-box performance
   constraint (`docs/adr/0005-stdlib-first-single-dependency.md`)?
5. **Fernet's own limitations.** Fernet is authenticated but all-or-nothing —
   FIX-03's chunked-sealing plan for large media needs either chunk-wise Fernet
   frames with an integrity-checked index, or an explicit documented size cap
   until a streaming AEAD construction is reviewed. Should that be folded into
   this review's scope now, or does it warrant its own follow-up review once
   FIX-03's plaintext-path streaming lands first (as FIX-03 itself proposes)?
6. **Migration safety** (§4, "backward-read, forward-write"). Does the proposed
   migration path introduce any downgrade or oracle risk during the window
   where both envelope forms are legal on read?
